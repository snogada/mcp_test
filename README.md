MCP 사용이유.
1. LLM의 초기 등장 (2020년 ~ 2022년)
 * 상황: GPT-3와 같은 모델들이 화려하게 등장했습니다.
 * 문제점: **"지식의 폐쇄성"**과 **"할루시네이션(환각)"**이 가장 컸습니다. 모델은 학습된 데이터 외에는 알지 못했고, 최신 정보나 사용자의 개인 데이터에 접근할 수 없었습니다.
 * 해결책: 이를 극복하기 위해 RAG(검색 증강 생성) 방식이 도입되어 외부 문서를 읽어오기 시작했습니다.
2. 에이전트와 도구 활용의 시작 (2023년 초)
 * 상황: LLM이 단순히 글을 쓰는 것을 넘어, 웹 검색이나 계산기 사용 같은 '행동'을 하기 원했습니다.
 * 문제점: **"파편화된 연결 방식"**이 문제였습니다. 서비스마다 API 규격이 다 달라서, AI에게 구글 캘린더를 읽히려면 구글용 코드를 짜고, 슬랙을 쓰려면 슬랙용 코드를 따로 짜야 했습니다. 개발 비용이 너무 높았죠.
 * 해결책: 'Function Calling(함수 호출)' 기능이 모델에 내장되기 시작하며 외부 도구와의 연결 가능성이 열렸습니다.
3. MCP의 탄생 배경 (2024년 말 ~ 현재)
 * 상황: 기업들은 수천 개의 데이터 소스(SQL, 로컬 파일, GitHub, SaaS 등)를 AI에 연결하고 싶어 합니다.
 * 문제점: **"N:M 연결의 늪"**에 빠졌습니다.
   * 호환성: 새로운 AI 모델이 나올 때마다 모든 데이터 소스를 다시 연결해야 합니다.
   * 보안: AI가 데이터에 접근할 때 보안 권한을 관리하기가 매우 까다롭습니다.
   * 복잡성: 데이터 소스를 교체하면 AI 시스템 전체를 수정해야 했습니다.
4. MCP(Model Context Protocol)라는 해결책
앤스로픽(Anthropic)이 주도하여 발표한 MCP는 이 모든 복잡함을 해결하기 위한 **'범용 플러그'**입니다.
| 구분 | 과거 방식 (Ad-hoc) | MCP 방식 |
|---|---|---|
| 연결 방식 | 서비스마다 개별 커스텀 코드 작성 | 표준 프로토콜로 한 번에 연결 |
| 확장성 | 모델이나 데이터가 바뀌면 전체 수정 | MCP 서버만 갈아 끼우면 즉시 작동 |
| 보안 | 로컬 데이터 접근이 어렵고 위험함 | 안전하게 로컬/원격 데이터를 모델에 전달 |
요약하자면 이렇습니다
 * 초기: AI가 아는 게 없어서 공부(데이터 학습)를 시켰습니다.
 * 중기: AI가 최신 정보를 알게 하려고 책(RAG)을 쥐여줬습니다.
 * 현재: AI가 내 컴퓨터의 파일을 읽고, 데이터베이스를 쿼리하고, 툴을 직접 쓰게 하려다 보니 **"서로 말이 통하는 공통 규격(MCP)"**이 필요해진 것입니다.
이제 개발자들은 각기 다른 AI 모델을 위해 삽질(?)할 필요 없이, MCP 서버 하나만 잘 만들어두면 어떤 최신 LLM이든 바로 우리 회사 데이터와 연결할 수 있는 시대가 된 것이죠.


 프로세스 흐름
​User Query: 사용자가 질문을 입력합니다.
​LLM Processing: LLM이 내장된 지식(Weights/Parameters)을 바탕으로 다음 단어를 예측합니다.
​Response: 학습 데이터 내의 정보만을 조합하여 최종 답변을 출력합니다.


​User Query: 사용자가 질문을 입력합니다.
​Search (Retrieval): 질문을 벡터(Vector)로 변환하여 외부 데이터베이스(Vector DB 등)에서 관련 문서를 찾습니다.
​Augment (Prompt Engineering): 질문과 검색된 문서 내용을 합쳐서 "이 정보를 바탕으로 답해줘"라는 식으로 프롬프트를 재구성합니다.
​LLM Processing: LLM은 주어지는 '참고 자료'를 읽고 답변을 생성합니다.
​Response: 근거가 명확한 최종 답변을 출력합니다.


### 🛠️ 랭체인 MCP 툴 등록 프로세스

이 그림은 랭체인이 어떻게 표준화된 MCP 인터페이스를 활용하여 외부 기능을 자기 것으로 만드는지를 보여줍니다.

1.  **발견 (Discovery):** 랭체인 앱이 MCP 서버에 연결하여 "어떤 기능을 제공하니?"라고 물어봅니다. (MCP 프로토콜 활용)
2.  **포장 (Wrapping/Definition):** MCP 서버가 제공하는 각 기능(예: `search_repositories`)을 랭체인의 `Tool` 객체로 자동 변환합니다. 이때 기능의 이름, 상세한 설명(Prompt), 필요한 인자(Schema) 정보가 함께 포장됩니다. **(그림의 'MCP Tool' 부분)**
3.  **등록 (Binding):** 이렇게 만들어진 MCP 툴들을 LLM에게 "너는 이제 이런 도구들을 쓸 수 있어"라고 알려주며 바인딩(bind)합니다.
4.  **실행 (Execution):** 사용자가 질문하면, LLM은 등록된 MCP 툴의 설명을 보고 필요한 도구를 선택하여 실행을 요청합니다.

1. stdio (Standard Input/Output)
​가장 기본적이고 널리 쓰이는 방식입니다. 로컬 환경에서 실행되는 프로세스 간 통신에 최적화되어 있습니다.
​동작 방식: 호스트(예: Claude Desktop)가 MCP 서버를 자식 프로세스로 실행하고, 표준 입력(stdin)과 표준 출력(stdout)을 통해 JSON-RPC 메시지를 주고받습니다.
​장점: 설정이 매우 간편하며 별도의 네트워크 포트 개방이나 보안 인증이 필요 없습니다.
​단점: 서버와 클라이언트가 반드시 동일한 기기에 있어야 합니다.
​용도: 개인용 로컬 도구, IDE 확장 프로그램.
​2. SSE (Server-Sent Events)
​웹 표준 기술을 사용하여 서버가 클라이언트로 데이터를 밀어넣어 주는 방식입니다. 원격 서버에 MCP를 구축할 때 주로 사용합니다.
​동작 방식: 클라이언트가 HTTP 연결을 맺으면, 서버는 연결을 유지한 채 실시간으로 이벤트를 보냅니다. 클라이언트에서 서버로의 요청은 별도의 HTTP POST 요청으로 처리됩니다.
​장점: HTTP 프로토콜을 그대로 사용하므로 방화벽 통과가 쉽고, 원격(Remote) 연결이 가능합니다.
​단점: stdio에 비해 구현이 복잡하며, 상태 유지(Stateful)를 위한 서버 자원 관리가 필요합니다.
​용도: 클라우드 기반 MCP 서비스, 다수의 사용자가 접속하는 중앙 집중형 도구.
​3. Streamable HTTP
​비교적 최근에 논의되거나 특정 프레임워크에서 강조되는 방식으로, 전통적인 SSE보다 유연한 양방향 스트리밍을 목표로 합니다.
​동작 방식: 단일 HTTP 연결 내에서 응답 본문(Response Body)을 끊지 않고 계속해서 데이터를 송수신하는 방식입니다.
​장점: SSE보다 오버헤드가 적을 수 있고, 현대적인 HTTP/2나 HTTP/3 환경에서 효율적입니다.
​단점: 인프라(프록시, 로드밸런서) 설정에 따라 연결이 강제로 끊길 수 있어 설정이 까다롭습니다.
​용도: 고성능이 필요한 원격 통신, 복잡한 비동기 작업 결과 전송.
